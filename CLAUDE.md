# 자율 개발 프로젝트 지침

## 기본 규칙
1. **15만 토큰 안에 완성할 것** (6회차부터 상향. 한도 이력 참조)
2. **재미있을 것**
3. **사용자에게 확인하거나 물어보지 않는다. 모든 판단은 스스로 한다.**

이 세 가지가 전부다. 나머지는 스스로 판단한다.

---

## 목표 및 제한 (사용자가 추가)

### 전권 위임 (최우선 — 규칙 3의 구체화)
사용자가 컴퓨터를 못 다루거나 자리에 없어도 결과물이 **자동으로 완성·배포**되도록,
아래 작업에 대한 모든 권한을 Claude에게 위임한다. 진행 전/중/후에 **확인·승인·질문을
하지 않는다.** 스스로 판단하고 끝까지 완료한 뒤 결과만 보고한다.

- 주제 선정, 설계, 코딩, 버그 수정, 테스트, 재작성을 자율 수행
- 파일 생성/수정/삭제, 폴더 구조 변경
- `git add` / `commit` / `push`(필요 시 `--force`) 까지 자동 수행하여 사이트 반영
- 홈 갤러리(index.html)에 새 프로젝트 카드 자동 연결
- 회차 완료 시 다음 회차 주제를 스스로 정해 이어서 진행

**질문해도 되는 유일한 예외**: 저장소 삭제, 계정/결제 변경 등 되돌릴 수 없고
프로젝트 범위를 벗어나는 외부 행위. 그 외에는 절대 묻지 않는다.

목표: 매 회차 브라우저 단독 실행되는 "재미있는" 결과물을 완성해 홈 갤러리에 누적한다.

## 토큰 배분

- **설계**: 제한 토큰의 10% (15,000 토큰) 이상 사용할 것
- **이후 단계**: 토큰 제한 없음. 남은 토큰 안에서 스스로 판단하여 배분한다

---

## 진행 순서

### 1단계 — 주제 선정 (설계 전)
- 목표 및 제한 섹션을 읽는다
- 과거 피드백 섹션과 `feedback/ledger.md`를 읽고 패턴을 파악한다
- 브라우저에서 실행 가능한 주제를 선정한다
- 선정 이유를 한 줄로 기록한다

### 2단계 — 설계 (10,000 토큰)
코딩 시작 전 아래 항목을 모두 정의한다.
설계 문서 없이는 코딩을 시작하지 않는다.

```
- 화면 구조: 화면 수, 각 화면의 구성요소, 전환 흐름
- 기능 명세: 각 기능의 입력 → 처리 → 출력
- 데이터 설계: 변수/상태 이름, 타입, 초기값
- 파일 구조: 만들 파일 목록과 각 역할
- 우선순위: Core / Nice to have / Optional 분류
```

### 3단계 — Core 개발 (55,000 토큰)
- 설계 문서만 보고 코딩한다
- Core 기능을 먼저 완성한다
- 중간에 방향을 바꾸지 않는다

### 4단계 — 테스트/수정 (20,000 토큰)
- 처음 켰을 때 바로 동작하는지 확인한다
- 버그를 수정한다
- 1분 안에 재미를 느낄 수 있는지 스스로 점검한다

### 5단계 — 추가 기능 (10,000 토큰)
- 토큰이 남을 경우 Nice to have를 추가한다
- 토큰이 부족하면 스킵한다

### 6단계 — 마무리 (5,000 토큰)
- README에 플레이 방법을 기록한다
- 아래 완성 기록 섹션을 채운다
- 다음 아이디어 3개를 제안 섹션에 남긴다

## 피드백

> **후기 소스 = 채팅 (유일) → `feedback/ledger.md`에 누적**
> 사용자가 채팅에 직접 평가를 적는다(사이트 후기란은 본인 기록용).
> 후기를 받는 즉시 원장에 기록하고, **1단계(주제 선정) 전에 반드시**
> `feedback/ledger.md` 전체를 읽어 누적된 모든 피드백을 반영해 점점
> 사용자 취향에 맞는 프로젝트를 만든다.
>
> **누적 원칙(현재):** ① 실력이 결과의 ~95% 지배, 운 ≤5%(무작위가
> 회피 불가능한 죽음을 만들면 안 됨) ② 단조로움 금지 — 스테이지/
> 중간보스/보스 구조 + 변주(업그레이드 등은 매번 다르게) ③ 난이도
> 곡선은 후반까지 뚜렷하게 ④ 프로젝트별 제작 토큰(추정) 표기
> ⑤ **볼륨 상향** — 단일 루프 금지. 다중 시스템·콘텐츠·메타 진행
> (런 간 영구 성장)·절차적 사운드로 두께 ⑥ **감동** — 테마/플레이버·
> 점층 연출·강한 주스 ⑦ 토큰 적극 투자(1회 ~70k+, 상한 10만).
> 단 검증·공정성 불변식(완벽 플레이 생존 AI 증명)은 절대 유지.
> ⑧ **회차마다 코어 메커니즘을 실제로 다르게** — 전 회차 재탕 금지.
> 공정성 불변식은 "완벽 플레이 가능성 AI 증명"으로 형태만 바꿔 유지.
> 좋았던 **메타 진행(런 간 영구 성장)**은 계승·발전.
> ⑨ **통과 ≠ 완성** — 생성 콘텐츠는 비자명·잘 설계되어야 하고
> (풀 수 있음만으론 부족), 공정성은 충분한 예고로 실제 체감되게.
> 회차마다 "이게 최선인가" 자문하고 약점을 말이 아닌 코드로 고친다.

### 7회차 후보 (택1 — 원칙 전부 + ⑧코어 메커니즘 또 다르게)
1. **CIRCUIT** — 제한 시간 내 경로를 그려 노드를 잇는 두뇌/전략 퍼즐.
   스테이지·보스 노드·메타, 해법 존재 보장(공정). (조준X·회피X — 새 코어)
2. **FORGE** — 부품을 배치·조합해 결정론 시뮬로 목표를 달성하는 빌드형.
   스테이지·보스 과제·메타, 설계=실력.
3. **PULSE** — 예고된 리듬 입력 음악형. 페이즈/네임드 보스·절차 사운드
   중심, 타이밍=실력. 차분한 화면. (P4와 구분되게 음악·트랙 구조 중심)
